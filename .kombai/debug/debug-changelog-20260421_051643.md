## Changes Made

### 1. Switched RelationMapper to Metadata-First Detection
- **File:** `components/RelationMapper.php`
- **Change:** Modified `buildForeignKeyConfig` to use the `DbTable` and `DbTableColumn` models as the source of truth for relationships instead of querying `INFORMATION_SCHEMA` from the physical database.
- **Reason:** Physical database foreign keys can become "broken" or point to temporary backup tables during the Sync process. Relying on metadata ensures the form builder always knows the intended relationship.
- **Revert:** Revert `buildForeignKeyConfig` to use `detectForeignKeys`.

### 2. Fixed Database Connection for Relation Discovery
- **File:** `controllers/FormController.php`
- **Change:** Updated `getForeignKeyConfigForForm` and `getRelationMapper` to pass the correct physical database connection for the specific project.
- **Reason:** When generating the form dropdown options, the system needs to query the actual project database (e.g., `sekolah`), not the application's metadata database.
- **Revert:** Remove the `$db` parameter passing in these methods.

### 3. Repaired Corrupted Physical Foreign Key
- **Database:** `sekolah` (Local MySQL)
- **Change:** Manually dropped the FK `fk_siswa_id_kelas` which was pointing to a deleted backup table and recreated it pointing to the active `kelas` table.
- **Reason:** This was causing empty dropdowns and potential submission errors.
